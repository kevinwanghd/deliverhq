# DeliverHQ 文档审计报告

**日期**: 2026-06-28  
**审计范围**: SKILL.md, AGENTS.md, README.md, references/*.md  
**审计框架**: 常见文档坏味道（借鉴软件工程最佳实践 + 推断的 Pocock 写作纪律）

## 五种文档坏味道（审计框架）

1. **Premature Completion（过早完工）**: 文档声称"已完成"，但实际是占位符/TODO。
2. **Duplication（重复）**: 同一事实在多处维护，导致不同步风险。
3. **Sediment（沉积）**: 过时信息未删除，与当前状态冲突（如已退役的功能仍在"可用"列表）。
4. **Sprawl（蔓延）**: 单一概念分散在多个文件，没有单一事实源（SSOT）。
5. **No-op（空转）**: 文档存在但不提供决策价值（纯重复代码注释、显而易见的内容）。

#### 问题 1.1: Sediment（沉积）— 过时的能力状态描述
**位置**: L63-65, L68-71  
**问题**: 
- "DesignGate / Orchestrator 按 experimental 使用" — 但 orchestrator 已在 5.10.0/5.11.0 稳定化（动词层是正式入口）。
- "Darwin Score / Quality Ratchet 已退役" — 这是对的，但放在"不进默认阻断链路"下容易误导（已退役 ≠ roadmap）。

**修复建议**:
```markdown
### 默认可用（5.12.0 起）

- CR 初始化和状态机：`init_cr.py` / `cr_state.py`
- 核心 Gate：SpecGate、DesignGate、ArchitectureGate、PreDevGate、ReviewGate、QualityGate、WritebackGate
- DevPhase Handoff：`dev_phase.py` 只做开发交接，不自动写码
- **用户面动词** (5.10.0+)：spec/design/dev/verify/archive（收口 54 个脚本的认知负荷）
- **动词路由器** (5.12.0+)：`route` 子命令推荐动词流（治 54→5 收口后剩余负荷）
- Gate Contract / selftest：验证脚本存在、参数契约和默认 pipeline 契约

### 需要谨慎

- DeployGate / PermissionGate：已接入但仍按 experimental 使用；PermissionGate 只是最小权限边界检查
- Worktree Manager：是工具，不是 Dev Agent；不要把它直接放进 pipeline 当开发阶段
- Failure Attribution / mistake-book dedup：基础链路已通，但规则演进仍需人工复核

### Roadmap（不承诺可用）

- Routing Eval JSON 化、Context schema、MaintenanceGate
- Dynamic Workflow / Tournament / Fan-out / Scout

### 已退役（`_archived/`，不要用）

- Darwin Score / Quality Ratchet / Loop Mode：AI 自评 / 自动循环执行违反核心哲学
```

**影响**: 中等（误导新用户 orchestrator 成熟度）

#### 问题 1.3: No-op（空转）— "何时使用"的❌列表过于显然
**位置**: L42-46  
**问题**: "不应该使用" 列表中的"只读分析、简单解释、文档错别字修复"是显而易见的（DeliverHQ 是重流程框架）。空转内容降低信噪比。

**修复建议**:
```markdown
❌ **不应该使用**：
- 紧急 hotfix（事后补文档可接受；如需实时治理，用 fast-lane 模式）
- 用户明确说"不要启动 DeliverHQ 流程"
- 一次性临时脚本（不涉及项目源码、无需留痕）

**判断启发**: 如果不需要留下可审计的工件链（spec/design/test evidence），大概率不需要 DeliverHQ。
```

**影响**: 低（UX 改进，非正确性）

#### 问题 2.1: Sprawl（蔓延）— Loop 可控性原则分散在多处
**位置**: 分散在 AGENTS.md、SKILL.md、references/loop-control.md  
**问题**: Loop 可控性（goal_contract/anti_gaming/retry_guard）的哲学、用法、动词集成分散在 3 个文件，没有明确的"从哪里开始读"锚点。

**修复建议**:
在 AGENTS.md 开头加：
```markdown
## Loop 可控性（防 Goodhart + 收敛出口）

DeliverHQ 的 loop 可控性由 3 件套组成，已集成进 `verify` 动词（5.11.0+）：
1. **goal_contract**: metrics + invariants 双轨校验（防"删测试让测试绿"）
2. **anti_gaming**: 反钻空子检查（信证据不信声明）
3. **retry_guard**: 重试上限 + 只读 status（失败后展示收敛状态，不自动 record）

**详见**: `references/loop-control.md`（设计哲学、用法、陷阱）  
**集成**: `verify` 动词链：`goal_contract*（条件） → review → quality → anti_gaming`
```

**影响**: 中等（提升可发现性）
