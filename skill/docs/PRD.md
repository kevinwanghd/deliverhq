> 本文是**产品意图的唯一来源**,仅人工维护意图部分;Agent 可以读取、校验、拆分和生成派生工件，但不能擅自改变产品意图。
> 例外:`writeback-agent` 可回填功能锚点的「关联 CR」行(该行不参与哈希)。
>
> **本文给人看，也要能被 Agent 稳定解析**:用于规划、评审、把握产品全貌；功能锚点中的 REQ/AC/INV/依赖/来源字段是结构化产品意图，不等同于实现代码。
> AI 干活使用本 PRD 派生的 `acceptance-spec.md`、`task-map.yml` 和 `prd-manifest.yml`。
> PRD 不直接承载代码实现细节；字段契约、接口 schema、测试步骤和禁改清单归 acceptance-spec。
>
> **单文件原则**:始终只用 `docs/PRD.md` 作为产品意图文件,不拆 `docs/prd/` 目录(避免双源)。
> 研发交付包可以在 CR 或外部交付目录中生成派生文件，但不得反向形成第二份产品意图。
> 产品很大时,靠锚点 ID 前缀分层(如 `[PRD-AUTH]` / `[PRD-AUTH-SSO]`),不拆文件。

# 第二部分:功能需求

> 每个 `## [PRD-XXX]` 章节 = 一个**功能锚点**,是 CR 的挂载点。
> **每个锚点章节单独参与哈希**(用于 PRD↔CR 对账),但章节内的「关联 CR」行**排除在哈希之外**。
>
> 锚点 ID 规则:
> - 写在标题里,如 `## [PRD-AUTH] 用户鉴权`
> - **只增不改**:ID 一旦使用永不修改(改了等于换锚点,挂载的 CR 全部失联)
> - 废弃:标 `## [PRD-AUTH] 用户鉴权 (已废弃)`,ID 保留
> - 老项目逆向生成的锚点标 `状态: reverse-engineered`,人确认后改为 `confirmed`
>   (这两个值是机器要识别的,保持英文)

## [PRD-TODO] 待办事项管理(示例锚点,与 CR-EXAMPLE 配套)

- **状态**: confirmed
- **优先级**: P1

**意图**
> 让用户用一份极简列表管理日常待办,把"想到/未做/已做"三类事项分得清,不被遗忘。

**用户故事**
> 作为日常使用者,我要快速增/查/改/删待办,以便随时把脑子里的事项卸载到工具里。

**功能描述**
> 一个最小可用的 CRUD:可以新增一条待办、查看分页列表、修改内容或状态、把不再需要的删除。删除是软删除——出于"误删可恢复"的考量,删除后不再出现在列表里,但底层保留。

**验收意图**(粗粒度,给人看)
> 增、查、改、删四件事都能做;查支持分页和状态筛选;非功能上,正常数据量下查询要够快、并发下不出错、API 文档要给开发者读。

**绝不能发生**(粗粒度负例,给人看)
> - 删除后还能在列表里查到(软删除应被列表过滤)
> - 列表泄漏其他用户的待办(将来加用户体系时的不变式)

**关联 CR**: CR-EXAMPLE
