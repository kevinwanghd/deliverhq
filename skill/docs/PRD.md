# 第二部分:功能需求

> 每个 `## [PRD-XXX]` 章节 = 一个**功能锚点**,是 CR 的挂载点。
> **每个锚点章节单独参与哈希**(用于 PRD↔CR 对账),但章节内的「关联 CR」行**排除在哈希之外**。
>
> 锚点 ID 规则:
> - 写在标题里,如 `## [PRD-AUTH] 用户鉴权`
> - **只增不改**:ID 一旦使用永不修改(改了等于换锚点,挂载的 CR 全部失联)
> - 废弃:标 `## [PRD-AUTH] 用户鉴权 (已废弃)`,ID 保留
> - 老项目逆向生成的锚点标 `状态: reverse-engineered`,人确认后改为 `confirmed`
>   (这两个值是机器要识别的,保持英文)

## [PRD-TODO] 待办事项管理(示例锚点,与 CR-EXAMPLE 配套)

- **状态**: confirmed
- **优先级**: P1

**意图**
> 让用户用一份极简列表管理日常待办,把"想到/未做/已做"三类事项分得清,不被遗忘。

**用户故事**
> 作为日常使用者,我要快速增/查/改/删待办,以便随时把脑子里的事项卸载到工具里。

**功能描述**
> 一个最小可用的 CRUD:可以新增一条待办、查看分页列表、修改内容或状态、把不再需要的删除。删除是软删除——出于"误删可恢复"的考量,删除后不再出现在列表里,但底层保留。

**验收意图**(粗粒度,给人看)
> 增、查、改、删四件事都能做;查支持分页和状态筛选;非功能上,正常数据量下查询要够快、并发下不出错、API 文档要给开发者读。

**绝不能发生**(粗粒度负例,给人看)
> - 删除后还能在列表里查到(软删除应被列表过滤)
> - 列表泄漏其他用户的待办(将来加用户体系时的不变式)

**关联 CR**: CR-EXAMPLE
